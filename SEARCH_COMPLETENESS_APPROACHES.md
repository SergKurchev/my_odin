# Подходы к оценке полноты поиска объектов на сцене

**Контекст задачи:**
ODIN-модель получает набор кадров сцены (RGB + depth) и сегментирует объекты (клубнику).
На данный момент модель не знает, все ли объекты на сцене уже видны в поступивших кадрах.
Нужен механизм, который даёт **сигнал уверенности**: «я нашла ВСЕ объекты» / «вероятно, есть скрытые объекты — нужно больше кадров».

---

## Подход 1: Вспомогательная голова «сцена завершена» (Scene Completeness Head)

### Идея
Добавить к ODIN отдельный классификационный/регрессионный выход, который принимает
агрегированное представление текущего набора кадров и предсказывает скалярную
вероятность `p_complete ∈ [0, 1]` — «все объекты найдены».

### Архитектура
```
[Pixel Decoder фичи] + [Маски текущих инстансов]
        │
   Global Average Pooling / Cross-frame Attention
        │
  FC(256) → ReLU → FC(1) → Sigmoid
        │
  p_complete
```

### Обучение
- **GT-метка**: `1`, если набор кадров покрывает ≥ 95% от общего числа объектов сцены (считается по аннотации всей сцены); `0` — иначе.
- **Loss**: BCE поверх основного panoptic loss:
  `L_total = L_panoptic + λ * BCE(p_complete, y_complete)`

### Модификация датасета
Для каждой сцены нужно:
1. Знать **полное** число объектов (из аннотации всей сцены, не отдельного кадра).
2. При генерации обучающих примеров случайно семплировать **подмножества кадров**
   (от 1 до N кадров из сцены) → метка = (найденных объектов / всего объектов) ≥ threshold.

### Плюсы
- Простота реализации (один дополнительный FC-блок).
- Явно обучаемый сигнал.

### Минусы
- Требует дорогой аннотации: для каждой сцены нужна полная 3D-аннотация всех объектов.
- Модель может переобучиться на артефакты (количество кадров, а не реальное покрытие).

## Подход 2: Маргинальный прирост новых инстансов (Marginal Instance Gain)

### Идея
Отслеживать, сколько **новых** объектов появляется при добавлении каждого нового кадра.
Когда прирост стабильно близок к нулю, система считает сцену «закрытой».

### Алгоритм (во время инференса, без дообучения)
```
newly_found = 0
for each new frame t:
    instances_t = ODIN(frames[0..t])
    delta = |instances_t| - |instances_{t-1}|
    newly_found = EMA(newly_found, delta, alpha=0.3)
    if newly_found < threshold:
        p_complete = HIGH  # стоп, все нашли
    else:
        p_complete = LOW   # нужен ещё кадр
```

### Плюсы
- Не требует изменений в датасете и архитектуре.
- Работает «из коробки» с текущей моделью.
- Интерпретируемо: человек видит, когда количество новых объектов падает.

### Минусы
- Ложное завершение: если скрытые объекты прячутся за другими и «новых» объектов давно нет,
  модель ошибочно решит, что сцена полная.
- Требует правильно подобрать `threshold` и `alpha` — эвристика, нет гарантий.
- Не учитывает геометрию сцены (слепые зоны).

## Подход 3: Геометрическое покрытие сцены (3D Occupancy Coverage)

### Идея
Строить инкрементальную **3D-карту занятости** (occupancy map) из глубинных кадров.
Предсказывать: какая доля «потенциально занятого» объёма уже наблюдалась хотя бы
с одного ракурса? Непокрытые плотные зоны = потенциально скрытые объекты.

### Алгоритм
1. Из depth-карты и позы камеры строить **точечное облако** каждого кадра.
2. Вокселизировать сцену (например, 3 см voxel).
3. Отмечать воксели как **наблюдаемые** (в cone зрения камеры) или **неизвестные**.
4. `coverage = наблюдаемые / (наблюдаемые + неизвестные)`.
5. Если `coverage > threshold` — сцена полная.

### Модификация ODIN
- Добавить модуль, который на основании позы камеры и карты глубины обновляет
  occupancy map и возвращает `coverage` как скалярный признак.
- Этот скаляр можно подавать в Scene Completeness Head (Подход 1) как дополнительный вход.

### Плюсы
- Физически обоснован: явно учитывает геометрию сцены и слепые зоны.
- Не требует полной аннотации объектов — нужны только позы камер и depth.
- Хорошо совместим с NBV-алгоритмом (тот тоже работает с occupancy).

### Минусы
- Требует known camera poses на момент инференса.
- Computationally expensive при больших сценах.
- Не отличает «пустое пространство» от «скрытый за объектом» — нужна дополнительная логика.

## Подход 4: Неопределённость модели через Monte Carlo Dropout (MC Dropout)

### Идея
Использовать **MC Dropout** (несколько прямых проходов с включённым dropout во время
инференса) для оценки эпистемической неопределённости модели. Если модель сильно
неуверена в масках на краях изображения или в occluded-зонах — значит, там могут быть
скрытые объекты.

### Алгоритм
```
T = 20 прямых проходов с dropout включён
для каждого пикселя:
    μ_mask = mean(masks по T проходам)
    σ_mask = std(masks по T проходам)

scene_uncertainty = mean(σ_mask по всем пикселям)
p_complete = sigmoid(-k * scene_uncertainty + b)
```

### Модификация ODIN
- Оставить dropout в трансформерном декодере **включённым** при инференсе.
- Добавить петлю из T проходов, агрегировать карту неопределённости.
- Порог `scene_uncertainty < threshold` → «сцена полная».

### Плюсы
- Не требует изменения датасета.
- Даёт **пространственную карту неопределённости** — видно, ГДЕ модель сомневается.
- Применимо к любой архитектуре с dropout.

### Минусы
- В T раз дороже по вычислениям.
- MC Dropout — приближённый байесовский метод, неопределённость может быть завышена
  в текстурированных зонах, не связанных со скрытыми объектами.
- Требует тонкой калибровки порогов.

## Подход 5: Self-supervised предсказание «скрытых» инстансов (Masked Instance Prediction)

### Идея
Обучить вспомогательную голову, которая смотрит на **частично наблюдаемые** кадры
(с окклюзиями) и предсказывает **количество** ненаблюдённых объектов. Аналог BERT-
маскирования, но для 3D-сцен.

### Обучение
1. Из сцены с N объектами случайно скрыть M объектов (убрать из обзора камер).
2. Подать в модель «неполные» кадры.
3. Целевая переменная: `M` (количество скрытых объектов) или бинарная метка `M > 0`.
4. Loss: MSE или Poisson regression по числу скрытых объектов.

### Архитектура
```
[Набор кадров] → ODIN encoder + decoder (заморожен)
     │
[Instance tokens + positional encoding]
     │
  Transformer (cross-attention по времени)
     │
  FC → predicted_hidden_count
```

### Модификация датасета
Для каждой сцены:
- Синтетически генерировать «неполные виды»: брать M < N объектов видимыми,
  остальные помечать как «скрытые».
- Это можно сделать программно из уже имеющихся 3D-аннотаций без дополнительной
  ручной разметки.
- `p_complete = P(predicted_hidden_count == 0)`.

### Плюсы
- Явно обучает модель понимать, чего **ей не хватает**.
- Датасет можно сгенерировать автоматически из существующих 3D-аннотаций.
- Предсказывает не просто «полная/неполная», а **сколько** объектов ещё не найдено.

### Минусы
- Требует значительной модификации пайплайна обучения.
- Модель может обучиться предсказывать число объектов по косвенным признакам
  (например, по размеру кустов), а не по реальной информации о скрытых объектах.

## Рекомендация по порядку разработки

1. **Начать с Подхода 2** (Marginal Instance Gain) — нулевые изменения в коде модели,
   позволяет сразу проверить базовую гипотезу.
2. **Параллельно готовить датасет** для Подхода 5 — автоматическая генерация «скрытых»
   инстансов из имеющихся 3D-аннотаций.
3. **Реализовать Подход 1 или 5** как основное решение после валидации Подхода 2.
4. **Подход 3** подключить как дополнительный сигнал в Completeness Head (Подход 1).
